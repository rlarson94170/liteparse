> **2026-06-04 inferred-tracks table detector — olmOCR-bench table_tests 38.8% → 39.5% (+0.7pp, +7 tests), Overall 35.1% → 35.2%; opendataloader & ParseBench unchanged.** Targeted the dominant `wrong_neighbors` failure on financial/earnings tables (e.g. Apple/Wiley `a769f9a3…pg2`, `fa18a15c…`, hundreds of similar narrow-numeric tables): the existing `try_detect_table` derives column tracks from the FIRST row's `split_cells` output, which uses `TABLE_CELL_GAP_FONT_MULTIPLIER × font_size` (= 14pt at 14pt body) as the inter-span gap threshold. The Apple Revenue Q3 row has `$448 → $427` and `$427 → 7%` gaps of 13.9pt — *just under* the threshold — so its 5 distinct PDFium items collapse to 3 cells (`$448 $427 7%` lumped). Every row in the Revenue table fails identically, so the table establishes as 3-col, and the cross-run merge can't recover (`merge_consecutive_table_runs` only handles `|A|+1 == |B|`).
>
> **New `try_detect_table_inferred` (`markdown_layout/tables.rs`).** Runs in `detect_tables` *before* `try_detect_table`. Steps:
>   1. **`infer_tracks_from_raw_items`**: walks forward from `start_idx` while `table_rows_adjacent` holds (cap `TABLE_TRACK_INFERENCE_MAX_ROWS=12`), collects `span.x` for every non-whitespace span on each row with ≥2 spans, then single-link clusters at `TABLE_TRACK_TOLERANCE_PT` (6pt). The Apple Q3 row alone exposes all 5 column anchors (49.5/119/152.6/186.2/253.3); even with PDFium-level merge on 9 Months row, the union across rows recovers the full geometry.
>   2. **Track-soundness guards** (all required): `tracks.len() ≥ TABLE_MIN_COLUMNS`, `tracks.len() > baseline_cells.len()` (only re-seed if we'd actually unlock more columns), and `min(inter-track gap) ≥ max(1.5 × seed_font_size, 12pt)` — rejects inferred "tracks" that are really inter-word positions in prose (word gaps are typically <1× font size).
>   3. **Strict seed-row check**: the seed (`lines[start_idx]`) must have `≥ tracks.len()` raw PDFium spans AND each span must single-cover (no track ranges where one span x-extent overlaps ≥2 anchors). This is the load-bearing false-positive guard — see "Seed strictness" below.
>   4. **`cells_from_raw_items_with_tracks`** per row: walk raw items, assign each to its nearest track within tol; for items whose `[x, x+width]` covers multiple tracks AND whose `x` anchors at the leftmost covered track (within tol), generalize `split_merged_at_anchor` to N anchors — pick whitespace positions whose linearly-interpolated x is closest to each subsequent track. Rejects rows with <2 PDFium spans (single-span lines spanning the track region are prose, not merged-numeric rows). Returns `Some(cells)` of length `tracks.len()` (some cells may be empty for sparse rows).
>   5. **Cell-text dedup via existing absorb path**: re-uses `absorb_header_lines` and the bold-first-row promotion already in `try_detect_table`. CV check via `row_spacing_cv` rejects runs with irregular cadence and falls back to the existing path.
>
> **Seed strictness was load-bearing.** The first cut required only `seed_spans + 1 ≥ tracks.len()` (one multi-cover split allowed in seed). That preserved +16 table tests but regressed multi_column by -5 — the failure was 2-column prose like `027c39…page_2` where PDFium emits "in the" and "study. The subjects including 343 males and 293" as two spans on one baseline. The first span at `x=56.7` width-extent reaches past `x=84.2` (the second span's anchor), so my detector saw "in the" as covering tracks 0 AND 1, split it at whitespace, and shredded the line into `| in | the study. The subjects... | males and 293 |` — a fake table. Tightening to `seed_spans ≥ tracks.len()` AND no multi-cover spans in the seed eliminated the regression (multi_column held exactly at 49.4%, no `order` tests lost) while still catching Apple's Q3 row (5 spans, all single-cover at 5 tracks). The trade: 9 of the 16 initially-won table tests were lost (cases where the table's first row had merged spans), but the +7 retained tests are net positive with zero collateral.

>
> **Root cause.** On 2-column papers a *centered full-width line* (authors / affiliation) sits above the body inside one region leaf. Its x-extent straddles the gutter, so the density V-cut sees no valley and can't split the columns; `build_projected_lines` then merges col1+col2 on shared baselines (e.g. "Abstract—This paper considers the problem of designing **surface in the presence of bounded** …"). The H-cut that would *peel* the spanning line exists, but `xy_cut_rec`'s `XY_MIN_LINES_PER_H_SIDE` guard rejects it because the spanning line is a single line, and the depth-0 banner bypass doesn't apply mid-recursion. Critically, the spanning line is often *just under* the 60% banner-width threshold (the IEEE authors line measured 290pt vs a 303pt threshold), so it's not even classified as a banner — the peeling H-cut has a **finite** density score.
>
> **Fix (`projection.rs`).** Added a `spanning_rescue` branch to the min-lines guard in `xy_cut_rec`: when one side of an H-cut is a single line (`thin`) and the other side (`wide`) genuinely splits into columns (`column_cut_x` — density V-cut OR `xy_find_column_cut` histogram returns a gutter x), AND the thin line's x-extent *crosses* that gutter, allow the cut despite the min-lines guard. The "thin line crosses the gutter" test is the discriminator that distinguishes a column-straddling banner (peel it) from a single-column caption in running prose (leave it) — a caption sitting inside one column never crosses the revealed gutter. Not gated on infinite banner score, so it catches the sub-banner-width case. New helpers `column_cut_x` (returns the gutter position) and `x_extent`. 2 new unit tests (`spanning_line_above_columns_is_peeled_to_reveal_gutter`, `single_column_caption_is_not_peeled_into_columns`); 168 lib tests pass.

> **2026-06-03 cross-run table merging — bench-neutral but structural cleanup.** Added `merge_consecutive_table_runs` post-pass to `detect_tables` in `markdown_layout/tables.rs`. Targets the dominant table_test failure bucket (~252 wrong_neighbors out of 589 non-image table failures): when `try_detect_table` breaks the run at a column-count change, the adjacent runs are merged in two cases. **Case Same**: identical column count + per-column track centers within `TABLE_TRACK_TOLERANCE_PT` → concat rows, keep A's header. **Case Subset**: `|A| + 1 == |B|` and A's tracks are a column-subset of B's tracks under a looser `TABLE_SUBSET_TRACK_TOLERANCE_PT = 12pt` (header cells often have different content widths than body cells) → fold A as B's header layers, joined per-column top-to-bottom with adjacent-dedup, padded with empty at the missing B-column index. > **Detection signal:** position band (top 15% OR bottom 15% of page) + isolation gap (≥ 1.0 × body_size to the nearest non-chrome-pattern neighbor in y-order) + chrome pattern signature. Patterns covered in `matches_chrome_pattern`: URL/DOI prefixes (`http://`, `https://`, `www.`, `doi:`, `doi.org/`, `dx.doi.org`), journal banner phrases (`please cite this article`, `contents lists available at`, `available online at`, `downloaded from`), copyright (`©`, `copyright `, `all rights reserved`), `Page N`/`Page N of M`/lone-digit page numbers (≤4 digits), `Vol.`/`Vol`/`Vol,` + digit (ASCII byte-scan, not regex), and journal-cite-style (4-digit year `19xx`/`20xx` + numeric `N-M` / `N–M` / `N—M` page range + ≤120 chars). 200-char ceiling on candidate strip so a body paragraph that happens to start with a URL isn't nuked. > **2026-06-02 sparse-new-row path in borderless detector.** Targeted the "first data row gets eaten by the header" failure mode (doc 180, 045, 088/089/090, 127, 083, 084 …). Old logic in `try_detect_table` (`markdown_layout/tables.rs`) only had one branch for partial-cell lines — a wrap-merge that fired any time `centroid_dy ≤ 1.5 × line_height AND all_align_track`. That ate any data row whose cell count was less than the header's into the header itself, fragmenting tables with sparse columns.
> - **New branch** runs *before* the wrap path: when (a) all cells map to distinct column tracks, (b) `cells.len() >= 2`, and (c) `bottom_gap >= 0.5 × line_height` (clear inter-row spacing, not a stacked continuation), promote the partial line to a new row with empty `TableCell`s at the missing track positions.

With it, 1-cell partial lines always fall through to the existing wrap-merge path (which is what we want — those are real continuations). The first attempt without this guard regressed TEDS by 0.04 across docs 130/051/082/090/147; adding it converted the net into +0.049.
> - **`bottom_gap >= 0.5 × line_height` discriminates "clear new row" from "wrap"** — a wrap typically overlaps the prior baseline (negative gap) or sits within ~0.3 × line_height of it. The 0.5 floor leaves a dead-zone (0.3, 0.5) that falls through to the existing wrap path, which is correct for the in-between cases.
> - **Wrap path unchanged.** First attempt tightened the wrap path's gap condition simultaneously, which regressed previously-clean tables (doc 130 0.84 → 0.10) because the existing wrap-merge was load-bearing for tightly-stacked multi-line cells. Reverted and reordered — sparse-new-row runs first, wrap path runs second with original conditions.

> **2026-05-31 XY-cut layout decomposition.** Replaced the flat x-density histogram column detector with a recursive modified XY-cut. Motivation: the old detector failed on layouts where the gutter is interrupted by a wide element (figures straddling both columns, full-width headings/banners) — every page-1-style case in the eval set, and the open carry-forward "column interleaving on page 1" from 2026-05-30.

> - **Types** (`types.rs`): added `Region` / `RegionKind { Leaf, Split }` / `CutAxis`. `ProjectedLine.column_id: usize` → `region_path: Vec<u16>` (index trail from root; equality = same leaf, prefix = containment). `ParsedPage.regions: Region` added behind `#[serde(skip)]` so JSON/text outputs stay byte-identical.
> - **Algorithm** (`projection.rs::xy_cut`): char-weighted 1D projection onto each axis at 2pt buckets. A "valley" is a contiguous run of buckets where density < `T_DENS × median(non-zero buckets)`. > **2026-06-01 column-start histogram fallback in XY-cut.** Targets the column-interleaving failure mode behind most remaining MHS=0 docs (decimal section heads on 2-column policy/academic/business reports — docs 029, 038, 039, 077, 189, …). The density-based V-cut search inside `xy_find_best_cut` fails on these layouts because a column-spanning element (wide heading, figure, table) fills the gutter at one y-band, inflating the density at the gutter's x and erasing the valley. Added `xy_find_column_cut` as a *fallback* consulted only when **both** `xy_find_best_cut` calls (H and V) return `None`.
>
> **Algorithm.** Histogram every item's left-edge x with 5pt buckets across the full item span (`min_x..max_x`, not bbox-relative — see the bbox aside below). Cluster contiguous occupied buckets (allowing ≤2-bucket gaps for intra-column drift) into peaks; each peak carries its summed item count. Then apply four conservative gates *in this order*:
> - `XY_COLUMN_MIN_LINES_PER_PEAK = 10` — drop peaks with fewer than 10 items. Indented paragraph-starts and footnote markers produce 3–6 item peaks; real column-left edges have 15+ items in a typical academic page.
> - **Exactly 2 strong peaks remaining.** 3+ peaks means tabular structure (one peak per cell column); we bail rather than slice a table.
> - `XY_COLUMN_PEAK_DOMINANCE = 0.55` — the two peaks together must hold ≥55% of all items. Rules out scattered-x prose layouts (lists, lab manuals, multi-indent docs) where the two tallest peaks are tall only relative to a flat tail.
> - `XY_COLUMN_PEAK_BALANCE_RATIO = 0.4` — smaller-peak / larger-peak count ratio ≥ 0.4. Filters out "single column with one off-x bullet stack" cases (90/10 split → false-positive column cut).
>
> Plus the existing geometric constraint: gap between the two peaks must be ≥ `XY_COLUMN_MIN_GAP_FRACTION = 0.25` × total item-span width. Cut position is the midpoint between the two peaks; returned `CutCandidate` carries `score = 1e9` so it beats every density-derived cut but still loses to `xy_find_banner_cut`'s `f32::INFINITY`.
>
> **Why histogram items, not lines.** The natural first cut was: y-band items into lines, take each line's `x_min`, histogram those. That fails on the *exact* layouts we're trying to fix — items from the left and right columns at the same baseline fold into one "line" whose `x_min` is the left-column edge and whose `x_max` reaches the right-column tail. The right column's left edge is hidden. Switching to per-item left edges lets each column's items contribute to their own peak directly.

> **2026-06-02 ruled-grid single-column tables (`tables.rs`).** Targeted the largest TEDS=0 cluster (15 docs with GT tables we emitted 0 columns for). Single-col GT table was 1 doc (149), but the same code path also unblocked thin-border-strip docs whose true content column was wedged between two phantom rule-strip columns. Three coordinated changes in `build_ruled_table`:
>
> - **Threshold relaxation:** `xs.len() >= 3` → `xs.len() >= 2`. Allows the detector to produce 1-column tables when the geometry only gives a single content column between borders.
> - **Phantom-row collapse:** after cell assignment, drop rows whose height < 0.8 × median non-empty row height AND that have no text. Doc 149 draws each visual row as three stacked rects (top-border 1pt, body 22pt, bottom-strip 5pt) which all survive 2pt y-clustering as separate grid rows — without collapse, the empty-cell fraction kills the table. Threshold preserves real fill-in-the-blank forms (empty body rows match median non-empty height).
> - **Phantom-column collapse:** mirror of the row rule for columns. Drop empty columns whose width < 0.3 × median non-empty column width. Doc 149 has 5.5pt left/right border-strip columns next to a 305pt content column — both drop cleanly.
>
> One stricter rule in `merge_table_runs`: a 1-column ruled run **yields to** a multi-column borderless run that overlaps the same line range. Without this, doc 078's data table (drawn with top/bottom/header rules but no internal V-rules) gets detected as a degenerate 1-col grid that overrides the correct 7-col borderless result. Path geometry strictly beats text alignment for *multi*-col ruled runs, but a 1-col ruled run is ambiguous — could be a real single-column table or a multi-col table missing its V-rules — so we defer to the multi-col borderless evidence when present.
